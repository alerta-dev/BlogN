Introduction to ROS

What is ROS?

ROS is an open-source robotics framework that makes it easier for the many moving parts in a robot to communicate with each other.

ROS also has a package system, which allows for easy sharing of code, reducing development time.

How does ROS work?

ROS can be defined as a communication protocol, a defined set of rules for how things communicate with each other. These rules can be overwhelming at face value, but to make it easier to understand, here's an analogy of how it works.

Analogy: Liquid Transportation in a Factory

Imagine an imaginary factory where there are stations that hold liquids and transfer liquids to other stations through pipes, using pumps and gauges.

Liquid

Liquid that flows through a pipe can be water, orange juice, hot chocolate, or any type of liquid. Any one pipe can only have one type of liquid moving through it.

Pipe

Some things to note about the pipe:

  • Liquid flows in one direction
  • Pipes can be "forked", so the liquid can flow to multiple destinations

Pumps and Gauges

Pumps actually move the liquid through these pipes, with their goal being to move the liquid to another station.

Gauges allow us to answer questions like What liquid is moving through this pipe? How much liquid is there?

Pumps and gauges are only found in stations.

Station

Some things to note about the station:

  • Can have mutiple pipes connected to it, and each pipe can either have the liquid flow in or out of the station
  • Pipes that have liquid flowing in have gauges and pipes that have liquid flowing out have pumps

Comparing the Analogy to ROS

Here is the mapping of terminology from the analogy to ROS:

  • Liquid -> ROS message
  • Pipe -> ROS Topic
  • Pumps and gauges -> ROS Publishers and Subscribers
  • Station -> ROS Node

Here is the analogy converted with ROS terminology, so you can go back to the analogy and compare if you get confused later.

Message

Messages are sent through a topic. Here is a list of common messages used. Any one topic can only have one type of message sent over it.

Topic

Some things to note about the topic:

  • ROS Messages flows in one direction on a topic
  • Topics can have multiple publishers that publish messages over it and also have multiple subscribers that subscribe to messages sent over it

Publishers and Subscribers

Publishers send messages over a topic, with their goal being to send the data in a message to another node.

Subscribers allow us to capture the data in a message that is sent over a topic and manipulate it using code to answer questions like: How fast should my robot go? How far should this servo turn?

Publishers and subscribers are initialized in nodes.

Node

Some things to note about the node:

  • Can have more than one publisher or subscriber initialized
  • One python file or C++ file usually has one node that does a particular function

ROS Example

We need to build a manual control node called manual_control that has 1 subscriber that subscribes to joystick messages of type sensor_msgs/joy on topic /joy and has 1 publisher that publishes velocity messages of message type std_msgs/Twist over topic /cmd_vel.
